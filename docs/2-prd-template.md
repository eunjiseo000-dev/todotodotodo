# PRD(Product Requirements Document) 템플릿

## ToDoToDoToDo 프로젝트용

## 📋 개요

이 문서는 ToDoToDoToDo 프로젝트의 **Product Requirements Document(PRD)** 작성을 위한 포괄적인 템플릿입니다.

**목적:**

- 기술 관점의 도메인 정의서를 보완하는 비즈니스/사용자 중심 문서
- 모든 이해관계자(PM, 개발팀, 디자인팀, QA팀)가 공유할 수 있는 명확한 요구사항 정의
- 프로젝트의 성공을 정량적으로 측정할 기준 제시

**사용 대상:**

- Product Manager (주 작성자)
- Engineering Lead (설계 검토)
- Design Lead (UX/UI 기준)
- QA Lead (테스트 계획)

**관련 문서:**

- [1-domain-definition.md](./1-domain-definition.md) - 기술적 도메인 정의 (참조 필수)

- 🔴 **어려움** - 2시간 이상, 팀 협의 필요

## Executive Summary

**난이도:** 🟡 **중간**
**우선순위:** ⭐⭐⭐ **필수**
**작성 시간:** 약 1시간

### 목적

- PRD의 핵심을 1-2페이지로 요약
- 경영진, 투자자, 새로운 팀원들이 프로젝트를 5분 안에 이해하도록
- 세부 섹션들의 "왜(Why)"를 먼저 제시

### 포함할 내용

```markdown
## Executive Summary

### 1. 한 줄 정의 (Elevator Pitch)

"[타겟 사용자]가 [기존 문제]를 해결하기 위해 [핵심 차별성]을 제공하는 [제품 카테고리]"

### 2. 문제 정의

- 사용자가 겪는 현재 문제점 (3-4가지)
- 문제의 심각도 및 영향 범위
- 기존 솔루션의 한계

### 3. 해결 방안

- 제품이 제시하는 핵심 솔루션
- 차별화 포인트
- 예상되는 효과

### 4. 비즈니스 목표

- 주요 목표 (3-4개)
- 성공 지표 (정량적 KPI)

### 5. 타겟 사용자

- Primary Persona 한 문장 설명
- Secondary Personas 리스트

### 6. 출시 일정 (Roadmap)

- MVP 출시: [날짜]
- Phase 2: [날짜]
- 장기 비전: [설명]

### 7. 핵심 성공 요소 3가지

1. [요소1]
2. [요소2]
3. [요소3]
```

### ToDoToDoToDo 예시

```markdown
## Executive Summary

### 한 줄 정의

"직장인과 학생이 빠르고 간단하게 일정을 추적하기 위해
최소한의 인터페이스와 초고속 응답으로 다른 할일 도구와 차별화되는
웹 기반 할일 관리 애플리케이션"

### 문제 정의

1. **복잡한 UI**: Todoist, Notion 등 기존 도구는 단순 할일 관리에는 기능이 과하고 학습곡선이 높음
2. **느린 응답**: 모바일과 데스크톱 간 동기화 지연으로 빠른 작업 입력 불편
3. **비용 부담**: 고급 기능을 위해 월 5,000원 이상의 비용 발생
4. **데이터 불안심**: 실수로 삭제한 할일을 복구할 수 없거나 복구 프로세스가 복잡

### 해결 방안

- **단순성**: 필수 기능만 엄선한 미니멀 인터페이스 (3가지 탭만)
- **속도**: 평균 API 응답 500ms 이하로 즉각적 피드백
- **안전성**: 휴지통 기능으로 실수 복구 가능
- **무료**: 영구 무료 기본 기능 제공

### 비즈니스 목표

1. 개인 생산성 관리의 표준 도구 위상 확보
2. 6개월 내 월간 활성 사용자(MAU) 5,000명 달성
3. 사용자 만족도(NPS) 40점 이상 확보
4. 향후 팀 협업 기능으로 B2B 시장 확대

### 타겟 사용자

- **주 사용자**: 일일 20개 이상 태스크를 관리하는 직장인 (25-45세)
- **보조 사용자**: 과제와 시험 일정을 추적하는 학생, 프로젝트 기한을 관리하는 프리랜서

### 출시 일정

- **MVP 출시**: 2025년 12월 (6주)
- **Phase 2** (태그, 검색): 2026년 1월-2월
- **Phase 3** (모바일 앱): 2026년 3월 이후
- **장기 비전**: 100만 사용자 플랫폼으로 성장

### 핵심 성공 요소

1. **단순함**: UI/UX의 최소화로 러닝커브 제거
2. **속도**: 모든 작업의 즉각적 피드백
3. **신뢰성**: 데이터 안전성으로 사용자 안심 확보
```

### 검증 체크리스트

- [ ] 한 줄 정의가 명확한가?
- [ ] 문제와 해결책이 명확하게 연결되어 있는가?
- [ ] 성공 지표가 정량적인가?
- [ ] 5분 안에 읽을 수 있는가?
- [ ] 팀의 모든 구성원이 동의하는가?

## 섹션 1: 비즈니스 전략 (Business Strategy)

**난이도:** 🟡 **중간**
**우선순위:** ⭐⭐⭐ **필수**
**작성 시간:** 약 2시간

### 목적

- 개인의 데일리 체크리스트 관리

### 포함할 내용

### 1.1 비전 및 장기 목표

```markdown
### 비전 (Vision)

개인이 데일리로 사용할 Todo 체크리스트 관리를 제공한다.

#### 예시 작성 팁:

- "~를 표준이 되게 한다"
- "~가 더 이상 고통이 아니도록 한다"
- "~의 새로운 경험을 제공한다"

### 장기 목표 (Long-term Goals)

없음

#### 예시:

- 목표 1: 시장 인지도 달성
- 목표 2: 사용자 규모 확대
- 목표 3: 수익화 달성
```

**ToDoToDoToDo 예시:**

```markdown
### 비전

"개인 생산성 관리의 표준 도구가 되어,
누구나 자신의 할일을 간편하게 관리할 수 있는 세상"

### 장기 목표

1. **2026년**: 월간 활성 사용자 100,000명, NPS 50 이상 (개인 사용자 시장 리더 위상)
2. **2027년**: 팀 협업 기능 출시, B2B 시장 진입 (월 5,000팀 사용)
3. **2028년**: 프리미엄 기능 통한 수익화 시작 (월 수익 1억원 달성)
4. **2030년**: 투자 또는 인수 합병을 통한 성장 가속화
```

### 1.2 타겟 시장 (Target Market)

```markdown
### 주요 시장 (Primary Market)

-

### 보조 시장 (Secondary Markets)

- 향후 확대할 시장
- 각 시장별 특성

### 시장 경쟁 상황

- 주요 경쟁사
- 우리의 포지셔닝
- 진입 장벽
```

**ToDoToDoToDo 예시:**

```markdown
### 주요 시장

- **카테고리**: 개인 생산성 도구 (Personal Productivity Tools)
- **지역**: 국내 (한국) 중심
- **타겟**: 25-45세 직장인, 대학원생, 프리랜서
- **시장 규모**: 국내 생산가능인구 약 3,000만명 (추정)
- **성장률**: 연평균 15-20% (디지털 업무 증가)

### 보조 시장

1. **팀 협업 도구 시장** (2027년 이후)
   - 소규모 팀 프로젝트 관리
   - 스타트업, 중소 팀 중심
2. **교육 시장** (2027년 이후)
   - 학생 과제 관리
   - 교육기관 벌크 라이선스

### 시장 경쟁 상황

- **주요 경쟁사**: Todoist, Microsoft To Do, Google Tasks, Notion
- **우리의 포지셔닝**: "심플함과 속도의 완벽한 균형"
- **진입 장벽**: 낮음 (온라인 서비스라 빠른 진입 가능) → 따라서 차별화 중요
```

### 1.3 핵심 가치 제안 (Value Proposition)

```markdown
### 차별화 포인트 (Differentiation)

- 경쟁사 대비 우리의 고유한 장점 3-5가지
- 각 포인트의 근거 (기술, 가격, 사용성 등)

### 고객이 얻는 이점 (Customer Benefits)

- 시간 절약
- 스트레스 감소
- 비용 절감
- 생산성 향상
```

**ToDoToDoToDo 예시:**

```markdown
### 차별화 포인트

1. **최고의 단순성**

   - 필수 기능만 집중: 추가 & 완료 & 삭제
   - 경쟁사(Todoist): 30개 이상 기능
   - 우리: 10개 이하 핵심 기능
   - 결과: 신규 사용자도 5분 내에 시작 가능

2. **초고속 응답 (평균 500ms 이하)**

   - 경쟁사: 2-3초 (특히 모바일에서)
   - 우리: 0.5초 이하
   - 결과: 지연감 없는 즉각적 피드백

3. **완전 무료 기본 기능**

   - 경쟁사: 기본 기능도 유료화 (월 4,900원~)
   - 우리: 모든 핵심 기능 영구 무료
   - 결과: 장기간 비용 걱정 없이 사용

4. **데이터 안전성 (휴지통 기능)**
   - 경쟁사: 휴지통 미제공 또는 기한 제한
   - 우리: 무한정 휴지통 보관 및 복원 가능
   - 결과: 실수 복구로 안심감 확보

### 고객이 얻는 이점

1. **시간 절약**: 복잡한 UI 학습 없이 5분 내에 시작
2. **스트레스 감소**: 빠른 응답으로 업무 흐름 방해 최소화
3. **비용 절감**: 모든 기본 기능 무료 (월 0원)
4. **심리적 안정감**: 휴지통으로 실수 복구 가능
5. **생산성 향상**: 간단한 우선순위 관리로 집중도 증대
```

### 1.4 수익 모델 (Revenue Model)

```markdown
### Phase 1: 사용자 확보 (현재 ~ 2026년 6월)

- 모든 기본 기능 무료 제공
- 목표: 월간 활성 사용자(MAU) 50,000명

### Phase 2: 프리미엄 기능 도입 (2026년 7월 ~)

- 기본 기능: 무료
- 프리미엄 기능: 월 4,900원
  - 고급 분석 대시보드
  - API 접근권
  - 우선 지원
- 예상: 프리미엄 전환율 10% → 월 50,000명 × 4,900원 × 10% = 월 2억원

### Phase 3: B2B 솔루션 (2027년 ~)

- 팀/조직 협업 버전
- 월 29,900원 (팀당)
- 목표: 5,000팀 × 29,900원 = 월 15억원

### 현재 단계: Phase 1 (사용자 확보 집중)

- 수익화 미룬 이유: 사용자 기반 확보 우선
- 수익화 시점: MAU 10,000명 이상 도달 시
```

## 섹션 2: 사용자 분석 (User Analysis)

**난이도:** 🔴 **어려움**
**우선순위:** ⭐⭐⭐ **필수**
**작성 시간:** 약 3-4시간

### 목적

- "누가 이 제품을 사용할 것인가?"에 대한 구체적 정의
- 모든 디자인, 기능 결정의 기준
- 사용자 행동을 예측하고 우선순위 결정

### 포함할 내용

### 2.1 Primary Persona (주요 사용자)

```markdown
### [Persona 이름]

#### 기본 정보

- 이름 (또는 대표 이름)
- 나이 / 성별
- 직업 / 직책
- 거주지역
- 월 소득 수준

#### 배경 (Background)

- 일일 업무량
- 현재 도구 사용 현황
- 기술 리터러시 수준
- 일일 스케줄

#### 목표 (Goals)

- 업무 목표
- 개인 목표
- 이 제품을 통해 달성하고 싶은 것

#### 문제점 (Pain Points)

- 현재 사용하는 도구의 불편함
- 극복하고 싶은 어려움
- 해결하고 싶은 문제 TOP 3

#### 이 제품에 기대하는 것

- 기대하는 기능
- 해결하길 원하는 문제
- 성공의 정의

#### 기술 숙련도

- 웹 기반 도구 사용 경험
- 모바일 앱 사용 정도
- 새로운 서비스 학습 의지
```

**ToDoToDoToDo Primary Persona 예시:**

```markdown
### Primary Persona: "바쁜 PM 영수"

#### 기본 정보

- 이름: 영수 (대표명)
- 나이: 35세 / 남성
- 직업: IT 회사 프로젝트 매니저
- 거주: 서울 강남구
- 월 소득: 600만원대

#### 배경

- 일일 업무 태스크: 25-30개
- 현재 도구: 엑셀 + 오래된 웹 기반 프로젝트 관리 도구 (느리고 복잡)
- 기술 리터러시: 높음 (매일 10개 이상 웹 도구 사용)
- 일일 스케줄:
  - 9:00-10:00: 회의 (태스크 우선순위 재조정)
  - 10:00-12:00: 각 팀원에게 업무 할당
  - 12:00-13:00: 점심시간 (중요 이메일 확인, 일부 업무 완료)
  - 13:00-17:00: 미팅 및 검토
  - 17:00-18:00: 내일 일정 정리, 오늘 완료 사항 확인

#### 목표

- 업무 목표: 프로젝트를 정시에 마감 내에 완료
- 개인 목표: 퇴근 후 개인 시간 확보, 스트레스 감소
- 제품 목표: 5분 내에 오늘 할일을 파악하고, 우선순위를 빠르게 재조정

#### 문제점 (TOP 3)

1. **시간 낭비**: 현재 도구에서 업무를 입력하고 확인하는 데 5-10분 소요

   - 아침 출근 시 어제 완료 상황 파악: 3분
   - 오늘 할일 입력/수정: 5-7분
   - 점심시간에 진행 상황 확인: 3-5분
   - 퇴근 전 내일 할일 정리: 5분
   - **일일 총 15-20분 낭비** → 월 5시간 낭비

2. **복잡한 UI**: 새로운 팀원이 도구 사용법을 배우는 데 30분 이상 소요

   - 복잡한 메뉴 구조
   - 잦은 업데이트로 변경사항 추적 어려움
   - 모바일 앱이 없어 진행 상황 확인 불편

3. **동기화 문제**: 데스크톱과 모바일(브라우저)에서 반영 지연
   - 모바일에서 수정한 내용이 데스크톱에 바로 안 보임
   - 새로고침해야 함
   - 실시간 협업이 어려움

#### 이 제품에 기대하는 것

- 필수 기능: 할일 추가 (3초), 완료 체크, 우선순위 변경 (드래그)
- 기대 기능: 모바일 웹 지원, 실시간 동기화
- 성공의 정의: "오늘 일정을 2분 내에 파악할 수 있다"

#### 기술 숙련도

- 웹 도구 사용: 매우 능숙 (Gmail, Slack, Jira, Asana 등 10개 이상 사용 중)
- 모바일 앱: 자주 사용 (카톡, 뱅킹 앱 등)
- 새 서비스 학습: 높음 (팀에 새 도구 도입 시 빨리 배우고 전파)
```


- [ ] 각 경쟁사와의 비교가 객관적인가?
- [ ] 우리의 차별화 포인트가 검증 가능한가?

## 섹션 4: 제품 기능 (Product Features)

**난이도:** 🟡 **중간**
**우선순위:** ⭐⭐⭐ **필수**
**작성 시간:** 약 1-2시간

### 목적

- 제품의 모든 기능을 명확하게 정의
- 각 기능의 우선순위를 MoSCoW 방식으로 분류
- 도메인 정의서와의 명확한 연결

### 포함할 내용

### 4.1 기능 목록 (MoSCoW 우선순위)

```markdown
## MoSCoW 우선순위 분류

   - 설명: [기능 설명]
   - 도메인 정의서 참조: [섹션 참조]
   - 개발 복잡도: [쉬움/중간/어려움]
   - 예상 개발 기간: [시간]

2. [기능명]
   [동일 형식]

### Should Have (중요 - 있으면 좋음)
```

**ToDoToDoToDo 기능 목록 예시:**

```markdown
## MVP (Phase 1) - 필수 기능

### Must Have (MVP에 필수)

#### 1. 회원가입 (FR-001)

- 설명: 이메일과 비밀번호로 새 계정 생성
- 도메인 정의서: 1-domain-definition.md 섹션 4.1
- 개발 복잡도: 중간
  - 이메일 중복 검증
  - 비밀번호 정책 검증 (8자 이상, 영문/숫자/특수문자)
  - bcrypt 암호화
  - 이메일 인증 (선택)
- 예상 개발 기간: 4시간

#### 2. 로그인 (FR-002)

- 설명: 이메일/비밀번호로 기존 계정 접근
- 도메인 정의서: 1-domain-definition.md 섹션 4.1
- 개발 복잡도: 중간
  - JWT 토큰 생성
  - 세션 관리
  - 로그아웃 기능
- 예상 개발 기간: 3시간

#### 3. 할일 조회 (FR-004)

- 설명: 사용자의 모든 할일 목록 표시
- 도메인 정의서: 1-domain-definition.md 섹션 4.2, 4.3
- 개발 복잡도: 쉬움
  - 필터링: 진행중, 완료, 휴지통
  - 정렬: 우선순위 순서로
  - 페이지네이션: (100개 이상 시)
- 예상 개발 기간: 2시간

#### 4. 할일 추가 (FR-005)

- 설명: 새로운 할일을 데이터베이스에 추가
- 도메인 정의서: 1-domain-definition.md 섹션 3.2, 5.1
- 개발 복잡도: 중간
  - 제목: 1-500자 검증
  - 시작일/종료일: YYYY-MM-DD 형식 검증
  - 시작일 ≤ 종료일 검증 (BR-001)
  - 우선순위 자동 할당 (마지막 값 + 1)
- 예상 개발 기간: 3시간

#### 5. 할일 수정 (FR-006)

- 설명: 기존 할일의 정보 변경
- 도메인 정의서: 1-domain-definition.md 섹션 3.2, 5.1
- 개발 복잡도: 중간
  - 할일 소유자 확인 (BR-002)
  - 속성 일부 업데이트 허용
  - 수정 시간 기록
- 예상 개발 기간: 2시간

#### 6. 할일 삭제 (FR-007)

- 설명: 할일을 휴지통으로 이동 (논리적 삭제)
- 도메인 정의서: 1-domain-definition.md 섹션 5.3
- 개발 복잡도: 쉬움
  - isDeleted 플래그를 true로 변경
  - deletedAt 타임스탬프 기록
  - 휴지통에서 조회 가능
- 예상 개발 기간: 1시간

#### 7. 할일 복원 (FR-008)

- 설명: 휴지통의 할일을 다시 활성화
- 도메인 정의서: 1-domain-definition.md 섹션 5.3
- 개발 복잡도: 쉬움
  - isDeleted 플래그를 false로 변경
  - 원래 우선순위로 복원
- 예상 개발 기간: 1시간

#### 8. 할일 영구삭제 (FR-011)

- 설명: 휴지통의 할일을 물리적으로 완전 삭제
- 도메인 정의서: 1-domain-definition.md 섹션 5.3
- 개발 복잡도: 쉬움
  - 데이터베이스에서 레코드 완전 제거
  - 확인 다이얼로그 필수 (실수 방지)
  - 감사 로그 기록
- 예상 개발 기간: 1시간

#### 9. 완료 체크 (FR-009)

- 설명: 할일의 완료 여부 토글
- 도메인 정의서: 1-domain-definition.md 섹션 5.4
- 개발 복잡도: 쉬움
  - isCompleted 플래그 토글
  - 완료 → 완료 목록으로 자동 이동
  - 미완료 → 진행중 목록으로 자동 이동
- 예상 개발 기간: 2시간

#### 10. 할일 순서 변경 (FR-010)

- 설명: 드래그 앤 드롭으로 우선순위 순서 변경
- 도메인 정의서: 1-domain-definition.md 섹션 5.5
- 개발 복잡도: 중간
  - 드래그 앤 드롭 API
  - priority 값 재계산
  - 다른 할일들의 priority 자동 조정
- 예상 개발 기간: 4시간

#### 11. 로그아웃 (FR-003)

- 설명: 사용자 세션 종료
- 도메인 정의서: 1-domain-definition.md 섹션 4.1
- 개발 복잡도: 쉬움
  - JWT 토큰 삭제/무효화
  - 세션 종료
- 예상 개발 기간: 1시간

### Should Have (v1.0 이후 우선 추가)

#### 1. 할일 검색 (FR-012)

- 설명: 할일 제목으로 검색
- 개발 복잡도: 쉬움
- 예상 기간: 1시간
- Phase: v1.1 (2026년 1월)

#### 2. 태그/카테고리 (FR-013)

- 설명: 할일을 카테고리별로 분류
- 개발 복잡도: 중간
- 예상 기간: 3시간
- Phase: v1.1 (2026년 1월)

#### 3. 반복 작업 (FR-014)

- 설명: 매일/매주/매월 반복되는 할일 설정
- 개발 복잡도: 어려움
- 예상 기간: 6시간
- Phase: v1.2 (2026년 2월)

### Could Have (v1.2 이후 고려)

#### 1. 할일 공유 (읽기 전용)

- 설명: 다른 사용자에게 할일 목록 읽기 권한 공유
- Phase: v2.0 (2026년 6월) - 팀 협업 기능 출시 시
- 이유: MVP에는 개인용으로만 설계

#### 2. 통지/알림

- 설명: 기한 1일 전 알림 발송
- Phase: v2.0 (2026년 6월)
- 이유: 우선순위 낮음 (사용자 피드백 후 추가)

### Won't Have (이번에 진행하지 않음)

#### 1. 팀 협업 기능

- 설명: 여러 사용자가 함께 할일 관리
- 이유: MVP는 개인용으로 집중, Phase 3에서 추가 예정
- 언제: v2.0 (2026년 6월 이후)

#### 2. 모바일 네이티브 앱

- 설명: iOS/Android 전용 앱
- 이유: 웹 기반 반응형으로 먼저 검증 필요
- 언제: v2.0 (2026년 6월 이후)

#### 3. 파일 첨부

- 설명: 할일에 파일 첨부
- 이유: 단순성 원칙 준수, 구글드라이브 등 외부 연동으로 충분
- 언제: 미정 또는 아예 미제공

#### 4. API 공개

- 설명: 서드파티 개발자를 위한 API
- 이유: 사용자 기반 충분해질 때까지 미연기
- 언제: v3.0 (2027년 이후)
```
